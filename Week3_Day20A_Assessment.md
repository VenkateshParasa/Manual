### Section A: MCQ Answers

1. **b) To provide consistency, quality assurance, and compliance**
   - Standards ensure consistent processes, proven practices, and regulatory compliance

2. **c) ISO/IEC Joint Technical Committee**
   - ISO/IEC JTC maintains the 29119 standard

3. **c) 5 parts**
   - Parts: Concepts, Processes, Documentation, Techniques, Keyword-Driven

4. **b) Concepts and Definitions**
   - Part 1 provides common vocabulary for testing

5. **d) Static Test Process**
   - The three levels are: Organizational, Management, and Dynamic

6. **d) 8**
   - Functional Suitability, Performance, Compatibility, Usability, Reliability, Security, Maintainability, Portability

7. **b) Security**
   - Security characteristic focuses on protecting information and data

8. **b) (Total Defects / Lines of Code) Ã— 1000**
   - Defect density is typically measured per 1000 lines of code

9. **c) Level 3: Compliant**
   - Level 3 indicates formal adoption with consistent application

10. **c) 12-18 months**
    - ROI typically becomes positive after 12-18 months

11. **b) Pass/Fail Criteria**
    - Pass/Fail Criteria is one of the 15 mandatory sections

12. **b) Degree to which product provides functions that meet stated needs**
    - Functional Suitability measures if functions meet requirements

13. **b) Specification-Based (Black-Box)**
    - EP is a black-box technique

14. **c) Keyword-Driven Testing**
    - Part 5 standardizes keyword-driven test approach

15. **c) Time Behavior**
    - Time Behavior sub-characteristic measures response times and throughput

**3. ISO 25010 Quality Requirements for Online Banking:**

**1. SECURITY (Critical)**
- Requirement: All data encrypted at rest (AES-256) and in transit (TLS 1.3)
- Test Approach: Penetration testing, vulnerability scanning
- Acceptance Criteria: Zero critical security vulnerabilities
- Priority: Critical

**2. RELIABILITY (Critical)**
- Requirement: 99.9% uptime (max 8.76 hours downtime/year)
- Test Approach: Stress testing, failover testing, soak testing
- Acceptance Criteria: System remains operational under peak load
- Priority: Critical

**3. PERFORMANCE EFFICIENCY (High)**
- Requirement: Transaction processing <2 seconds (95th percentile)
- Test Approach: Load testing with JMeter (10,000 concurrent users)
- Acceptance Criteria: Response time <2s, throughput >100 TPS
- Priority: High

**4. FUNCTIONAL SUITABILITY (Critical)**
- Requirement: 100% accuracy in financial calculations
- Test Approach: Precision testing, reconciliation testing
- Acceptance Criteria: Zero calculation errors, all transactions match
- Priority: Critical

**5. USABILITY (Medium)**
- Requirement: First-time users complete transfer in <3 minutes
- Test Approach: Usability testing with 10 users
- Acceptance Criteria: 80% success rate, SUS score >70
- Priority: Medium

**6. COMPATIBILITY (High)**
- Requirement: Works on Chrome, Safari, Firefox, Edge (latest 2 versions)
- Test Approach: Cross-browser testing
- Acceptance Criteria: 100% functionality on all browsers
- Priority: High

**4. ISO 29119-3 Mandatory Test Plan Sections:**

**5 Mandatory Sections:**

1. **Test Plan Identifier**
   - Unique ID for the test plan
   - Example: TP-BANK-2026-Q1-v1.0

2. **Test Items**
   - What will be tested
   - Example: Payment module, Transfer module, Account management

3. **Features to be Tested**
   - Specific features in scope
   - Example: Fund transfer, Bill payment, Account statements

4. **Pass/Fail Criteria**
   - Specific metrics for success
   - Example: 100% P0 passed, 95% P1 passed, zero critical defects

5. **Test Deliverables**
   - Documents and reports to be produced
   - Example: Test plan, test cases, execution reports, defect reports

**Additional Mandatory Sections:**
6. Test Approach
7. Suspension & Resumption Criteria
8. Test Environment
9. Responsibilities
10. Schedule
11. Risks & Contingencies
12. Approvals

**Keywords:** identifier, items, features, criteria, deliverables, approach, environment, responsibilities, schedule, risks

**5. ISO 29119 Three Process Levels:**

**1. ORGANIZATIONAL TEST PROCESS**
**Scope:** Entire organization
**Purpose:** Define high-level test policy and strategy

**Activities:**
- Define organizational test policy
- Establish test strategy
- Conduct test process assessments
- Drive continuous improvement

**Example:**
"ACME Corp establishes a policy that all software must achieve 85% code coverage
and pass security scans before production release. This applies to all 50 development
teams across the organization."

**Artifacts:** Test Policy, Test Strategy, Process Improvement Plan

---

**2. TEST MANAGEMENT PROCESS**
**Scope:** Individual test project
**Purpose:** Plan, monitor, and control testing for a specific project

**Activities:**
- Create test plan for the project
- Monitor test execution progress
- Control and adjust based on results
- Complete testing and document lessons learned

**Example:**
"For the Mobile Banking App v2.0 project, the Test Manager creates a test plan
covering 500 test cases, assigns 5 QA engineers, schedules 4 weeks of testing,
and tracks daily progress against the plan."

**Artifacts:** Test Plan, Progress Reports, Completion Report

---

**3. DYNAMIC TEST PROCESS**
**Scope:** Individual test execution
**Purpose:** Design, execute, and report on specific tests

**Activities:**
- Design test cases from requirements
- Set up test environment and data
- Execute tests and log results
- Report incidents (defects)

**Example:**
"QA Engineer designs 50 test cases for the fund transfer feature, sets up test
accounts with various balances, executes tests, and logs 3 defects found during
execution."

**Artifacts:** Test Cases, Test Data, Execution Logs, Incident Reports

**Keywords:** organizational, management, dynamic, policy, strategy, planning, monitoring, execution, artifacts, scope
